# Deploying the SMS

Target setup: **API on Render (free tier) · Web on Vercel (free tier) · Database on Neon (free tier)**.
Total cost: $0/month for a pilot. Free-tier caveat: the Render API sleeps after ~15 minutes of
inactivity, so the first request after idle takes ~30–60s. Upgrade the Render service to a paid
plan when a school starts using this daily.

## Prerequisites (accounts you need)

1. **GitHub** — both platforms deploy from a GitHub repo. Push this repository there first.
2. **Render** — https://render.com (sign in with GitHub).
3. **Vercel** — https://vercel.com (sign in with GitHub).
4. **Neon** — already set up; you need the connection string (Dashboard → your project → Connection string).

## 1. API on Render

1. Render dashboard → **New → Blueprint** → select this GitHub repo.
   Render reads `render.yaml` from the repo root and pre-configures the `sms-api` service.
2. When prompted for environment variables (the ones marked `sync: false`):
   - `DATABASE_URL` — the Neon connection string (must end with `?sslmode=require`)
   - `WEB_ORIGIN` — leave blank for now; fill in after step 2 below with the Vercel URL
   - `PAYSTACK_*`, `ARKESEL_*` — leave blank until you have (test) keys; payments/SMS
     features fail gracefully without them
   - JWT secrets are auto-generated by Render (`generateValue: true`) — don't set them by hand
3. Deploy. When it's live, note the URL (e.g. `https://sms-api.onrender.com`) and check
   `https://sms-api.onrender.com/health` returns `{"status":"ok"}`.

## 2. Web on Vercel

1. Vercel dashboard → **Add New → Project** → import the same GitHub repo.
2. Settings before the first deploy:
   - **Root Directory**: `sms/apps/web`
   - **Framework preset**: Next.js (auto-detected)
   - Vercel detects pnpm from the lockfile; no build-command override needed.
3. Environment variable:
   - `NEXT_PUBLIC_API_BASE_URL` = the Render API URL from step 1 (no trailing slash)
4. Deploy. Note the URL (e.g. `https://sms-web.vercel.app`).

## 3. Connect them

1. Back in Render → `sms-api` → Environment → set `WEB_ORIGIN` to the Vercel URL
   (e.g. `https://sms-web.vercel.app`). Multiple origins: comma-separate them.
   The service redeploys automatically.
2. Smoke test: open the Vercel URL → log in (`admin@sunrise.test` / `Admin123!` if the
   demo seed has been run against this database) → learners list should load.

## 4. Later: payment & SMS keys

- **Paystack** (https://paystack.com) — Ghana business account. Use **test keys**
  (`sk_test_…`) first; set `PAYSTACK_SECRET_KEY`, `PAYSTACK_PUBLIC_KEY`, and configure the
  webhook URL in the Paystack dashboard to `https://<render-url>/payments/webhooks/paystack`,
  then put the webhook secret in `PAYSTACK_WEBHOOK_SECRET`.
- **Arkesel** (https://arkesel.com) — SMS API key into `ARKESEL_API_KEY`; sender ID (needs
  NCA registration, Arkesel assists) into `ARKESEL_SENDER_ID`. Delivery-report callback URL:
  `https://<render-url>/messages/webhooks/arkesel/dlr`.

## Notes

- **Migrations** run automatically on every deploy (`prisma migrate deploy` in the start
  command) — they only apply committed migration files, never invent schema changes.
- **Seeding**: to load demo data into a fresh database, run locally with `DATABASE_URL`
  pointed at it: `pnpm --filter api run seed` then
  `npx ts-node -r tsconfig-paths/register src/prisma/demo-seed.ts` from `apps/api`.
- **Custom domain**: add it in Vercel (web) and Render (API) dashboards, then append the
  new web origin to `WEB_ORIGIN`.
- **Backups**: Neon free tier keeps limited history. Before real school data goes in,
  either upgrade Neon or schedule `pg_dump` backups — do not skip this.
